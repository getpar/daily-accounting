## 核心功能需求

### 1. 花销记录
- 记录每一笔花销的**金额**（单位：人民币/元）
- 记录花销的**日期和时间**
- 可选记录**备注**（自由文本描述）
- 支持对已记录的花销进行**查看、编辑、删除**

### 2. 花销分类（二级分类体系）

#### 一级大类 & 二级小类

| 一级分类 | 二级小类 |
|---------|---------|
| 🍜 **餐饮饮食** | 早餐、午餐、晚餐、零食饮料、外卖、聚餐聚会 |
| 🚗 **交通出行** | 公交地铁、出租车/网约车、加油充电、停车费、火车/高铁、飞机 |
| 🛒 **购物消费** | 服装鞋帽、数码产品、日用百货、美妆护肤、奢侈品类 |
| 🏠 **住房居家** | 房租/房贷、水电燃气、物业费、家具家电、维修保养、日用品 |
| 🎮 **娱乐休闲** | 电影演出、旅游度假、运动健身、游戏充值、KTV/酒吧 |
| 💊 **医疗健康** | 看病就诊、药品购买、体检保健 |
| 📚 **教育学习** | 培训课程、书籍资料、考试报名、文具用品 |

| 📦 **其他杂项** | 快递邮寄、宠物开销、其他 |

### 3. 数据概览
- 查看**当月总支出**
- 按**分类统计**支出占比
- 查看**每日支出趋势**

### 4. 数据持久化
- 数据**本地存储**，无需联网
- 数据可**导出备份**（Excel/CSV格式）

## 🚀 启动方式

### 开发模式（日常开发/测试用）

**桌面版：**
```bash
cd "d:\ClaudeDemo\记账APP\desktop"
npm run dev
```

**小程序版（需要 Node.js v20）：**
```bash
# 先切换到 Node v20（默认 v24 不兼容 Taro）
set PATH=%USERPROFILE%\node-v20\node-v20.18.0-win-x64;%PATH%
cd "d:\ClaudeDemo\记账APP\miniapp"
npm run build:weapp
# 然后用微信开发者工具打开 dist/ 目录
```

### 注意事项

**桌面版：**
- 如果启动时报 `Cannot read properties of undefined (reading 'whenReady')` 错误，说明环境变量 `ELECTRON_RUN_AS_NODE` 被设为了 `1`。需要先执行 `unset ELECTRON_RUN_AS_NODE`（Mac/Linux）或在系统环境变量中删除该项（Windows），再重新启动。
- 应用启动后会自动打开 Electron 窗口。
- 数据存储在 `%APPDATA%/daily-notes/daily-notes.db`（Windows）或 `~/Library/Application Support/daily-notes/daily-notes.db`（Mac）。

**小程序版：**
- ⚠️ **编译前必须先关闭微信开发者工具**，否则 `dist/` 目录被锁，新代码写不进去。
- 小程序使用 Taro 3.x，需要 Node.js v20（系统默认 v24 不兼容）。
- 数据存储在微信本地 Storage 中。

### 打包为可安装的 exe/dmg
```bash
npm run build
```
产物在 `release/` 目录下。

## 技术栈选型方案对比

> 💡 **背景说明**：考虑到你未来想把软件做成"微信小程序 + 手机App"，技术选型需要同时兼顾「现在能快速做出桌面端」和「未来能低成本扩展到小程序和手机端」。以下三个方案都满足 Windows + Mac 桌面运行。

---

### 方案一：🅰️ **Electron + React**

**通俗解释**：把网页技术（做网站的那套东西）打包成桌面软件。就像把网页"装进一个框里"，让它看起来用起来像一个真正的桌面软件。

| 维度 | 说明 |
|------|------|
| **桌面端** | ✅ Windows + Mac，一次编写即可 |
| **未来扩展到微信小程序** | ✅ 可用 Taro 框架把代码转成小程序（约70%代码可复用） |
| **未来扩展到手机App** | ✅ 可用 React Native 或 Taro 转成手机App |
| **开发速度** | ⚡ 快 — 技术成熟，现成轮子多 |
| **软件体积** | 🐘 较大（约80-150MB） |
| **运行性能** | ⭐⭐ 中等 — 吃内存较多（类似开一个浏览器） |
| **学习成本（对你未来找人维护）** | ⭐⭐ 低 — 会做网页的开发者很多，容易找到人接手 |
| **界面美观度** | ⭐⭐⭐ 高 — 可做出非常精美的界面 |

**优势总结**：
- 技术最成熟，遇到问题最容易搜到答案
- 做网页的开发者最多，未来容易找人维护或接手
- 扩展到微信小程序和手机App的路径最清晰

**劣势总结**：
- 软件体积较大，占用硬盘空间多
- 内存占用较高（相当于后台跑了一个精简版浏览器）
- 启动速度不如原生应用快

---

### 方案二：🅱️ **Tauri + React**

**通俗解释**：和方案一类似（用网页技术做界面），但"外壳"更轻量。方案一用的是"大而全的浏览器外壳"，方案二用的是"操作系统自带的最精简外壳"，所以做出来的软件体积小、跑得快。

| 维度 | 说明 |
|------|------|
| **桌面端** | ✅ Windows + Mac，一次编写即可 |
| **未来扩展到微信小程序** | ✅ 同样可用 Taro 框架转成小程序（和方案一相同） |
| **未来扩展到手机App** | ✅ 同样可用 React Native 或 Taro |
| **开发速度** | ⚡ 中等 — 较新，部分功能需要自己摸索 |
| **软件体积** | 🐁 很小（约5-15MB） |
| **运行性能** | ⭐⭐⭐ 好 — 内存占用低，启动快 |
| **学习成本（对你未来找人维护）** | ⭐⭐⭐ 较高 — 技术较新，懂的人相对少 |
| **界面美观度** | ⭐⭐⭐ 高 — 和方案一相同 |

**优势总结**：
- 软件体积非常小（比方案一小10倍以上）
- 运行流畅、省内存、启动快
- 使用的 Rust 底层语言安全性高

**劣势总结**：
- 技术比较新（2022年才发布），成熟度不如 Electron
- 懂这套技术的开发者较少，未来找人维护稍难
- 部分复杂功能可能需要自己动手解决（社区资源不如方案一丰富）

---

### 方案三：🅲 **Flutter**

**通俗解释**：Google 出品的一套"画 UI"技术，用一种编程语言写出 Windows、Mac、iOS、Android 全部平台的软件。它不是网页技术，而是直接在屏幕上"画"出界面。

| 维度 | 说明 |
|------|------|
| **桌面端** | ✅ Windows + Mac |
| **未来扩展到手机App** | ✅ iOS + Android 原生支持（最强项） |
| **未来扩展到微信小程序** | ⚠️ **最大问题** — 无法直接转换，小程序需要重新开发（代码复用率不到30%） |
| **开发速度** | ⚡ 中等 |
| **软件体积** | 🐘 中等（约30-60MB） |
| **运行性能** | ⭐⭐⭐ 很好 — 接近原生应用的流畅度 |
| **学习成本（对你未来找人维护）** | ⭐⭐ 中等 — Flutter 开发者数量在增长，但不如网页开发者多 |
| **界面美观度** | ⭐⭐⭐ 高 — Material Design 风格 |

**优势总结**：
- 桌面端和手机端代码几乎100%复用，第二期做手机App极快
- 性能流畅，接近原生应用体验
- Google 持续投入，生命力强

**劣势总结**：
- **微信小程序不支持 Flutter**：这是最大短板。如果未来确定要做小程序，小程序那部分需要另起炉灶重新开发
- 桌面端支持相对较新（2022年才稳定），可能有小坑
- 界面风格偏"Google 风格"，不太像传统 Windows/Mac 软件

---

### 📊 三方案对比总览

| 对比维度 | 方案A: Electron | 方案B: Tauri | 方案C: Flutter |
|----------|:---:|:---:|:---:|

### 🎯 我的推荐：**方案一 — Electron + React**

**推荐理由**：

1. **小程序是决定因素**：你明确提到未来要做微信小程序。方案三（Flutter）对小程序支持最差，基本排除。方案一和方案二在小程序扩展上能力相同，但方案一更成熟。

2. **你初次使用 Claude Code**：方案一技术最成熟，社区资源最丰富。遇到问题时，我帮你搜到的解决方案最多、最可靠。开发过程会更顺畅。

3. **未来找人接手最容易**：方案一使用的是全球最主流的网页开发技术（React），国内开发者数量极大。如果未来你想找开发者接手继续改，门槛最低。

4. **方案二的 Tauri 确实更优（软件小、性能好），但对于你的情况，方案一的"稳"比方案二的"轻"更重要。** 作为非技术用户 + 第一个项目，稳字当头。

## 第二轮技术选型：UI组件库 & 数据存储

> 💡 **背景说明**：确定了"用什么造房子"（Electron + React），接下来要决定两件事——"房子装修成什么样"（UI组件库）和"账本数据放在哪里"（数据存储方案）。

---

### 🎨 UI组件库 — 决定App长什么样

组件库就像一个"家具套装"——你不用自己从头打桌子做椅子，直接挑选现成的按钮、输入框、表格、弹窗来拼出你的软件界面。

#### 方案A：🅰️ **Ant Design** ⬅️ 推荐

**通俗解释**：蚂蚁金服出品的UI组件库，国内最流行。就像宜家家具——种类齐全、质量靠谱、说明书（文档）写得清楚。

| 维度 | 说明 |
|------|------|
| **视觉风格** | 简洁专业的企业风格，配色克制 |
| **中文支持** | ⭐⭐⭐ 完美 — 中文文档、中文组件、中文社区 |
| **组件丰富度** | ⭐⭐⭐ 极高 — 表格、表单、图表、日期选择器等应有尽有 |
| **适合场景** | 数据管理类应用（记账就属于这一种） |
| **上手难度** | ⭐ 低 — 文档详尽，示例多 |
| **界面美观度** | ⭐⭐⭐ 高 — 专业、干净、耐看 |

**优势**：
- 国内最主流，中文文档质量极高，搜问题容易找到答案
- 表格和表单组件特别强（记账App最核心的就是填表单 + 看数据表）
- 对数据展示、统计图表支持好（后续做支出统计直接能用）

**劣势**：
- 风格比较"企业化"，不太花哨（但对于记账软件来说反而是优点）
- 软件打包体积会因此增大一些

---

#### 方案B：🅱️ **shadcn/ui**

**通俗解释**：新一代的"精品定制家具"——不是整套买，而是一块一块挑选，每一块都经过精心设计，你拥有完全的控制权。

| 维度 | 说明 |
|------|------|
| **视觉风格** | 现代极简风，干净漂亮，设计感强 |
| **中文支持** | ⭐⭐ 一般 — 文档是英文的，但组件本身支持中文 |
| **组件丰富度** | ⭐⭐ 中等 — 基础组件齐全，但高级组件（图表等）需自己组合 |
| **适合场景** | 追求设计感的现代应用 |
| **上手难度** | ⭐⭐ 中等 — 需要一些额外配置 |
| **界面美观度** | ⭐⭐⭐ 最高 — 2024年最流行的设计风格 |

**优势**：
- 颜值最高，做出的软件看起来非常现代
- 代码完全属于你，修改起来很自由
- 在开发者社区口碑极好，热度高

**劣势**：
- 2023年才发布，相比Ant Design不够成熟
- 组件数量不如Ant Design多，比如没有自带的图表组件
- 文档是全英文的，对非技术用户查阅不便

---

#### 方案C：🅲 **Arco Design**

**通俗解释**：字节跳动（抖音/今日头条的母公司）出品的UI组件库。设计风格比Ant Design更年轻活泼。

| 维度 | 说明 |
|------|------|
| **视觉风格** | 年轻化、活泼，设计语言现代化 |
| **中文支持** | ⭐⭐⭐ 完美 — 中文文档，字节跳动团队维护 |
| **组件丰富度** | ⭐⭐⭐ 高 — 和Ant Design不相上下 |
| **适合场景** | 追求年轻、现代设计风格的应用 |
| **上手难度** | ⭐ 低 — 文档清晰 |
| **界面美观度** | ⭐⭐⭐ 高 — 设计感好，比Ant Design更活泼 |

**优势**：
- 设计语言比Ant Design更年轻时尚
- 中文文档质量好，字节团队持续维护
- 组件丰富度接近Ant Design

**劣势**：
- 社区规模和流行度不如Ant Design（用户量约为其1/5）
- 第三方教程和讨论比Ant Design少
- 更新频率不如Ant Design稳定

---

**🎯 我的推荐：方案A — Ant Design**

理由很简单：你的App核心功能是"填表单记账 + 看数据统计"，这正是Ant Design最擅长的场景。中文文档最好、社区最大、遇到问题最好解决。对于你的第一个项目，选它就是最稳的选择。

### 💾 数据存储方案 — 决定账本数据放在哪里

你的记账数据是核心资产，存哪里、怎么存，直接关系到**数据安全、查询速度、以及未来做多端同步时的难度**。

#### 方案X：🇽 **SQLite 数据库** ⬅️ 推荐

**通俗解释**：SQLite就像一个"迷你Excel数据库引擎"。你的记账数据被组织成一张张表格（就像Excel工作表），可以快速查找、统计、排序。全世界无数手机App和桌面软件都在用它。

**类比**：给你的账本配了一个专职会计——不管你有100条记录还是10万条，它都能快速帮你查到"上个月吃饭总共花了多少钱"。

| 维度 | 说明 |
|------|------|
| **数据容量** | ⭐⭐⭐ 百万级记录毫无压力 |
| **查询速度** | ⭐⭐⭐ 极快 — 尤其做统计汇总时 |
| **数据安全** | ⭐⭐⭐ 高 — 成熟的数据库，不会因意外关机损坏数据 |
| **未来做多端同步** | ⭐⭐⭐ 容易 — 主流同步方案都支持SQLite |
| **导出备份** | ⭐⭐⭐ 容易 — 可导出为各种格式 |
| **实现复杂度** | ⭐⭐ 中等 — 需要安装一个数据库模块 |

**优势**：
- 极其可靠，全球有上百万应用在使用，数据不会丢
- 查询统计飞快（算"本月餐饮总支出"这类操作瞬间完成）
- 为未来多端数据同步打好基础
- 支持复杂查询（比如"对比这三个月每月的交通支出趋势"）

**劣势**：
- 比直接存文件多一步配置
- 数据文件不是直接双击就能打开的（需要专门的工具查看）

---

#### 方案Y：🇾 **JSON文件存储**

**通俗解释**：数据直接存成电脑上的一个文件，格式简单，用记事本就能打开看。就像一个"纯文本账本"。

**类比**：用一个TXT记事本记账——每条记录写一行。简单直接，但如果记录多了，翻查统计就慢了。

| 维度 | 说明 |
|------|------|
| **数据容量** | ⭐⭐ 几千条记录没问题，上万条后变慢 |
| **查询速度** | ⭐ 慢 — 每次都把整个文件读进内存 |
| **数据安全** | ⭐⭐ 中等 — 写入时如果断电可能损坏文件 |
| **未来做多端同步** | ⭐⭐ 中等 — 需要自己处理数据冲突 |
| **导出备份** | ⭐⭐⭐ 直接复制文件即可 |
| **实现复杂度** | ⭐ 最简单 — 不需要额外模块 |

**优势**：
- 实现最简单，数据就是一个文件，直接复制就能备份
- 用记事本就能打开看原始数据
- 不需要安装任何额外的东西

**劣势**：
- 数据多了以后（超过几千条），统计会很慢
- 不如数据库可靠，意外关机会有小概率文件损坏
- 不能做复杂查询（比如"按月份和分类同时统计"就很难实现）
- 未来做手机同步时，几乎需要重写整个存储部分

---

#### 方案Z：🇿 **IndexedDB（浏览器内置数据库）**

**通俗解释**：这是每个浏览器（包括Electron内置的浏览器）自带的存储机制。就像浏览器里有一个"隐形的数据库"，不需要额外装任何东西。

**类比**：浏览器自带的保险柜——免费、随用随取、但功能不如专业数据库强大。

| 维度 | 说明 |
|------|------|
| **数据容量** | ⭐⭐ 中等 — 单个网站通常限制几百MB |
| **查询速度** | ⭐⭐ 中等 — 比文件快，但不如SQLite |
| **数据安全** | ⭐⭐ 中等 — 浏览器管理，应用程序层面控制力弱 |
| **未来做多端同步** | ⭐ 困难 — 不是为文件同步设计的 |
| **导出备份** | ⭐ 麻烦 — 需要专门写导出功能 |
| **实现复杂度** | ⭐⭐ 中等 — API比较复杂 |

**优势**：
- 不需要额外安装任何东西，Electron 自带
- 适合简单的数据存取

**劣势**：
- 操作API比较复杂，代码写起来绕
- 数据管理和备份不方便
- 未来扩展性差，基本只适合简单场景
- 统计查询能力弱

---

#### 📊 数据存储方案对比总览

| 对比维度 | X: SQLite | Y: JSON文件 | Z: IndexedDB |
|----------|:---:|:---:|:---:|
| 数据容量 | ⭐⭐⭐ 上百万条 | ⭐⭐ 数千条 | ⭐⭐ 中等 |
| 统计查询速度 | ⭐⭐⭐ 飞快 | ⭐ 慢 | ⭐⭐ 中等 |
| 数据安全可靠 | ⭐⭐⭐ 非常可靠 | ⭐⭐ 一般 | ⭐⭐ 一般 |
| 备份导出难度 | ⭐ 需写功能 | ⭐⭐⭐ 直接复制 | ⭐ 需写功能 |
| 未来多端同步 | ⭐⭐⭐ 容易 | ⭐⭐ 困难 | ⭐ 非常困难 |
| 开发复杂度 | ⭐⭐ 中等 | ⭐ 简单 | ⭐⭐ 中等 |

**🎯 我的推荐：方案X — SQLite**

这是三个方案中唯一一个能陪你走到底的。现在是单机记账，未来要做手机版 + 数据同步，SQLite 是唯一不需要"推倒重来"的方案。多花一点点前期功夫，换未来省大事。
